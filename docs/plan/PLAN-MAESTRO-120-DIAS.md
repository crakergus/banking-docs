## 🧭 Estrategia de 120 Días (vs 100)

### ¿Por qué 120 días?

El plan original de 100 días es viable pero frágil. 120 días agrega **3 semanas estratégicas** que convierten un plan agresivo en uno resiliente:

| Semana extra | Ubicación | Propósito |
|--------------|-----------|-----------|
| **Buffer 1** | S7 (fin Mes 1) | Consolidar fundamentos + catch-up + monolito pulido |
| **Buffer 2** | S14 (fin Mes 2) | Consolidar microservicios + limpiar deuda técnica + repaso AWS parcial |
| **Buffer 3** | S19-S20 | Semanas exclusivas de simulacros SAA intensivos + presentación final |

### Distribución de Fases

```
Fase 1 (S1-S7):    Fundamentos AWS + Clean/Hexagonal + Enterprise Foundations — 42 días
Fase 2 (S8-S14):   AWS Avanzado + Microservicios + Event-Driven + Enterprise Applied — 42 días
Fase 3 (S15-S18):  Resiliencia + Observabilidad + Seguridad + Enterprise Consolidation — 24 días
Fase 4 (S19-S20):  SAA Intensivo + Presentación Enterprise Final — 12 días
```

**Total:** 420h en 120 días hábiles (3.5h × 120)

## 🤖 Uso de IA en el Plan

### Principio: Regla 80/20
- **80% esfuerzo propio** — tú decides, diseñas, escribes el primer draft, te grabas hablando
- **20% IA como amplificador** — revisa tu trabajo, desafía tus decisiones, acelera boilerplate, corrige inglés

La IA NO reemplaza el aprendizaje. La IA ACELERA la ejecución una vez que entiendes el concepto.

### Herramientas IA por actividad

| Actividad | Herramienta | Cómo usarla | Cuándo NO usarla |
|-----------|-------------|-------------|-------------------|
| Escribir código del proyecto | Kiro (IDE con IA) | Scaffolding, boilerplate, tests, refactoring asistido | Para diseño arquitectónico — eso lo decides tú |
| Code review | Kiro | Detectar code smells, sugerir mejoras, validar patterns | Para aprobar tu propio código sin revisar |
| Revisar arquitectura | ChatGPT / Claude | Simular stakeholder questions, challenge decisions | Para que diseñe la arquitectura por ti |
| Speaking practice | ChatGPT Voice | Architecture review roleplay, Q&A, interview sim | Para reemplazar grabarte hablando solo |
| Corregir inglés escrito | ChatGPT / Grammarly | ADRs, PRs, docs — corrección + vocabulario técnico | Para escribir el contenido desde cero |
| Aclarar conceptos AWS | ChatGPT / Claude | Dudas puntuales, comparaciones rápidas de servicios | Para estudiar temas completos (usa Cantrill) |
| Generar preguntas SAA | ChatGPT | Preguntas de áreas débiles como complemento | Para reemplazar TutorialsDojo/SkillBuilder |
| Planificación semanal | Kiro / ChatGPT | Generar el detalle día a día, ajustar plan | Para decidir qué estudiar (eso ya está definido) |

### Reglas de uso por bloque

**🇬🇧 Inglés (1h 30min):**

## 🚀 Proyecto Base: Banking Platform (Banking Light)

### Visión del Producto
Plataforma modular cloud-native para fintechs y cooperativas pequeñas en LATAM. Core extensible con dominio banking como primer vertical. Diseñada para que en el futuro se puedan enchufar otros dominios (e-commerce, gobierno, seguros) sin modificar el core.

> **Nombre de trabajo:** `banking` (se renombrará con marca comercial post-120 días)  
> **Package Java:** `com.banking.*`  
> **Repos:** `banking-docs`, `banking-identity-service`, `banking-accounts-service`, etc.

### Arquitectura de Alto Nivel

```
┌─────────────────────────────────────────────────────────┐
│                    API GATEWAY                            │
├─────────────────────────────────────────────────────────┤
│                 PLATFORM CORE (agnóstico)                 │
├───────────────┬───────────────┬─────────────────────────┤
│  Identity &   │  Notifications│  Audit &                │
│  Entitlements │  (multi-canal)│  Compliance             │
├───────────────┴───────────────┴─────────────────────────┤
│            DOMAIN: BANKING LIGHT (enchufable)             │
├───────────────┬───────────────┬─────────────────────────┤
│  Accounts     │  Transactions │  Reports                │
│  (saldos,     │  (transfers,  │  (movimientos,          │
│   estados)    │   pagos)      │   estados de cuenta)    │
└───────────────┴───────────────┴─────────────────────────┘
```

### Módulos

| Módulo | Tipo | Responsabilidad |
|--------|------|-----------------|
| **Identity & Entitlements** | Core | Usuarios, roles, permisos, multitenancy, onboarding (KYC light) |
| **Accounts** | Domain | Cuentas (ahorro, corriente), saldos, estados, apertura |
| **Transactions** | Domain | Transferencias internas/externas, débitos, créditos, compensaciones |
| **Notifications** | Core | Email, SMS, push — configurable por tenant y evento |
| **Reports & Audit** | Core/Domain | Movimientos, estados de cuenta, audit trail inmutable |

### Evolución Arquitectónica (20 semanas)
```
S1-S2:    Monolito (Clean Architecture) — Identity + Accounts básico + ADRs + TOGAF intro
S3-S4:    Hexagonal Architecture + Entitlements completo + TOGAF Phase B
S5-S6:    Bounded Contexts + Deploy AWS (ECS + RDS + CI/CD)
S7:       🔄 BUFFER — Consolidación Fase 1 + AS-IS/TO-BE documentation
S8-S9:    Split a Microservicios (Identity, Accounts, API Gateway) + TOGAF Migration
S10-S11:  Transactions module (CQRS + Event-Driven) + TOGAF Change Management
S12-S13:  Transfer Saga + Outbox + Idempotencia + ARB simulation
S14:      🔄 BUFFER — Consolidación Fase 2 + Technology Radar + Katas pendientes
S15-S16:  Resiliencia + Observabilidad + Notifications module + NFR/SLI/SLO
S17:      Seguridad (OAuth2, tenant isolation, encryption, audit) + compliance
S18:      Enterprise Consolidation: Roadmap + Well-Architected Review
S19-S20:  🔄 SAA Intensivo + Presentación Enterprise Final
```

### Justificación de Patrones por Módulo

| Patrón | Dónde aplica | Por qué es natural |
|--------|-------------|-------------------|
| Clean Architecture | Identity (S1-S2) | Separar dominio de infra desde el inicio |
| Hexagonal | Entitlements (S3-S4) | Ports para diferentes auth providers |
| Microservicios | S8-S9 | Identity escala diferente que Transactions |
| CQRS | Reports (S10-S11) | Read-heavy (consultas de movimientos) vs Write (registrar transacción) |
| Event-Driven | Transactions → Notifications → Audit (S10-S11) | TransferCompleted → NotifyUser → LogAudit |
| Saga | Transfers (S12-S13) | Débito → Crédito → Confirmar (compensable si falla) |
| Outbox + Idempotencia | Transactions (S12-S13) | Garantizar exactly-once en transferencias |
| Circuit Breaker | External payment gateway (S15) | Integraciones externas pueden fallar |
| Feature Flags | Todo el sistema (S17) | Encender/apagar módulos por tenant |

### Dominio de Negocio: Flujos Principales

**1. Onboarding (KYC Light):**
```
RegisterUser → ValidateIdentity → CreateAccount → NotifyUser
Estados: PENDING → VALIDATED → ACTIVE | REJECTED
```

**2. Transferencia Interna (Saga):**
```
InitTransfer → ValidateBalance → DebitSource → CreditTarget → Confirm → Notify
Compensación: si CreditTarget falla → RevertDebit → NotifyFailure
```

**3. Consulta de Movimientos (CQRS):**
```
Write side: TransactionCreated event → stored en command DB
Read side: Projection actualiza vista optimizada de movimientos
Query: GET /accounts/{id}/movements?from=&to= (read DB)
```

### Extensibilidad futura (post Día 120)

| Nuevo dominio | Qué reutiliza del core | Qué agrega |
|---------------|----------------------|------------|
| E-commerce | Identity, Notifications, Audit, API Gateway | Products, Orders, Cart, Shipping |
| Gobierno (matrícula) | Identity, Notifications, Audit | Vehicles, Owners, Renewals, Payments |
| Seguros | Identity, Notifications, Audit | Policies, Claims, Underwriting |

El core se mantiene intacto. Solo se enchufan nuevos módulos de dominio.

## 🏗️ ¿Qué son las Architecture Katas?

Las **Architecture Katas** son ejercicios de diseño arquitectónico creados por **Neal Ford** (coautor de "Fundamentals of Software Architecture"). El nombre viene del concepto de "kata" en artes marciales: una forma que se practica repetidamente hasta dominarla.

### Formato
1. Te dan un **problema de negocio** con restricciones concretas (usuarios, SLA, budget, compliance)
2. Diseñas una **solución arquitectónica** completa en tiempo limitado
3. Documentas: diagrama C4, selección de tecnología, trade-offs, ADR principal

### Ejemplo
> "Diseña un sistema de notificaciones multi-canal (email, SMS, push) para 10M usuarios con entrega en <5s y budget $3K/mes."

### Para qué sirven en este plan
- Entrenan pensamiento arquitectónico ante problemas NUEVOS (no solo el Payroll)
- Practican comunicación de decisiones (como en una entrevista Staff/Principal)
- Se documentan en inglés (doble beneficio)
- Son el ejercicio más parecido a lo que hace un arquitecto en la vida real

### Diferencia con el proyecto Payroll
- **Payroll** = profundidad (un sistema, evolución completa)
- **Katas** = amplitud (problemas variados, diseño rápido)
